Level 2 Data Flow Diagram
Process selected: 2.0 — Manage Payments. Decomposed into five sub-processes.
Sub-processNameDescription / Data Flows
2.1Create PaymentFinance Officer supplies payment fields + supplier ID. Reads D1 to auto-fill TIN. Writes new record to D2 with status = Draft.
2.2Edit PaymentFinance Officer supplies updated fields + payment ID. Updates Draft or Ready record in D2. Validates for duplicate invoice number against D2.
2.3Delete PaymentFinance Officer supplies payment ID. System checks status = Draft only; deletes record from D2. Rejected if status ≠ Draft.
2.4View PaymentFinance Officer supplies payment ID or filter criteria. System reads D2 and D1; returns full payment detail including supplier info.
2.5Link to BatchFinance Officer supplies payment ID + batch command (add/remove). Updates batch_id on D2. Writes event to D6 (Audit Log). Enforces payment status transition rules.

Section 18
Group Reflection

1. Which model helped your group understand the system best?

The Entity Relationship Diagram was the most clarifying model. When we mapped entity relationships explicitly, we discovered that storing a single batch_id directly on the Payment table was insufficient for the "Remove Payment from Batch" use case. A payment moved between batches would silently overwrite its previous batch membership with no audit trail. This led us to introduce the BatchAuditLog entity — essential for a finance system operating under institutional audit requirements.

2. Which model was most difficult to develop?

The Level 2 DFD was the most challenging — specifically specifying exact data flows for sub-process 2.5 (Link to Batch), which crosses three data stores: D2 (Payment Data), D3 (Batch Data), and D6 (Audit Log). Deciding the appropriate level of detail without over-specifying implementation required several iterations. We also found that aligning the Level 2 DFD with the payment status lifecycle — clarifying which sub-process controls which status transition — was essential to remove ambiguity from the "controlling process" notation we initially used.

3. How did system modeling improve your project design?

Modeling forced implicit decisions to become explicit before any code was written. Specifically: the use case enumeration revealed that our first draft captured only 6 of 22 use cases, leaving the Finance Supervisor and System Administrator interactions entirely undesigned. The payment status lifecycle table resolved a group disagreement about whether a payment removed from a batch returns to Draft or Ready status. The sequence diagram for UC-04 clarified that the duplicate invoice check should be scoped per supplier, not globally. Writing the constraints section made us realize the system must operate offline, which directly informed our technology choice of a local Apache/PHP/SQLite stack rather than a cloud-dependent architecture.
