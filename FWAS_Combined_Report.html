§ 02

Concept Building

Activity 2.1 — Understanding Architecture

Software Architecture

Software architecture is the high-level structure of a software system — the set of principal design decisions that define how the system is decomposed into components, how those components interact, and what properties the overall system must satisfy. For FWAS, the architecture defines how the web interface, the business logic layer, and the SQLite database are organized and how data flows between them.

Architectural Design

Architectural design is the process of making and documenting those high-level structural decisions before detailed coding begins. It translates requirements into a blueprint that development teams can implement consistently. For FWAS, this meant deciding on a layered architecture, selecting PHP as the backend language, and defining how each layer communicates.

Why Architecture Bridges Requirements and Design

Requirements describe what the system must do. Architecture answers how those requirements will be realized structurally. It is the bridge because it forces abstract needs to become concrete structural choices: a requirement for offline operation became the decision to use a local Apache/PHP/SQLite stack; a requirement for role-based access became a dedicated authentication layer.

Justification

A layered architecture was chosen because FWAS has a clear separation between what the user sees, what the system enforces, and what is stored. This separation was not arbitrary — it emerged from the Week 3 modeling work. When the group discovered the BatchAuditLog requirement during the ER diagram exercise, it was because the data layer had a concern entirely separate from the presentation layer. Layering made that separation explicit and enforceable.

§ 06

Application Architectures

⚡
System Type Selected: Transaction Processing System (TPS)

FWAS is a TPS. Its primary function is to capture, validate, and record discrete financial transactions and produce outputs that feed into an external system (IFMS). Every key interaction — recording a payment, adding it to a batch, finalizing and exporting — is a transaction with defined pre-conditions, validation rules, and state changes. The payment status lifecycle (Draft → Ready → Batched → Processed) is characteristic of TPS design.

System NameFinance Workflow Assistance System (FWAS)

7.2 Component Description

ComponentRole
AuthControllerValidates login credentials against bcrypt-hashed passwords. Initializes session with user ID and role. All other controllers check session first.
SupplierControllerCRUD for supplier records. Validates TIN (exactly 10 numeric digits). Provides TIN auto-fill to PaymentController on supplier selection.
PaymentControllerCreates and manages payment records. Enforces status lifecycle. Validates for duplicate invoice numbers per supplier.
BatchControllerCreates and manages batches. Enforces that only Ready payments are added. Transitions Open → Finalized. Writes every add/remove to BatchAuditLog.
DocumentControllerHandles file upload, storage, and retrieval for payment-attached documents.
ExportControllerReads finalized batch data joined with supplier and payment records. Produces CSV or Excel file for manual IFMS entry.

7.3 Technology Stack

LayerTechnology
FrontendHTML5, CSS3, vanilla JavaScript — rendered server-side by PHP. No framework dependency.
BackendPHP 8.1 on Apache HTTP Server (XAMPP). Plain MVC — no framework — for offline reliability.
DatabaseSQLite 3 via PHP PDO. Schema: Supplier, Payment, Batch, Document, User, BatchAuditLog.
ExportPHP native CSV + PhpSpreadsheet library for .xlsx output.
Securitybcrypt via password_hash(). Session-based auth. Role-based access in every controller.

7.4 Non-Functional Requirements

QualityHow FWAS Ensures It
SecurityPasswords as bcrypt hashes — never plaintext. Session gates all routes. BatchAuditLog is append-only with no exposed delete method.
PerformanceAll processing is local — no network round trips. SQLite performs well for single-office sequential load. TIN auto-fill removes repeated lookup overhead.
ScalabilityIntentionally scoped to single institution. Future migration from SQLite to PostgreSQL/MySQL requires only PDO config changes — the layered architecture isolates this.
§ 08

Group Reflection — Week 5

1. Most Challenging Concept

The most challenging concept was correctly applying the 4+1 architectural views without conflating them. Initially the group confused the Process View (runtime behavior) with the Development View (code structure). The distinction became clear when we asked: "Is this describing what the code looks like on disk, or what happens when it runs?" The physical view was also non-trivial — we had to be precise that deployment is genuinely local with no cloud component, which is architecturally significant given the offline requirement.

2. Architectural Decision the Group Struggled With

The group debated whether to use Laravel versus plain PHP MVC. Laravel would have provided scaffolding, routing, and ORM features, but it introduces Composer dependency management and requires internet access during setup — incompatible with the offline deployment requirement. We resolved this by choosing plain PHP MVC, accepting more manual code in exchange for a stack that installs and runs entirely offline from a USB drive if necessary.

3. How We Would Improve the Design

Three v2 improvements would be prioritized: (1) a formal Supervisor approval workflow — requires adding an Approval state to the Batch enum and an ApprovalController; (2) direct IFMS API integration to eliminate the manual retype step, subject to government infrastructure availability; (3) automated database backup scheduling via a PHP cron job writing timestamped SQLite copies to a backup folder.

§ K

Design Patterns in Practice

Activity K1 — The Design Problem

A significant coupling problem emerged in how different parts of FWAS respond when a Payment record changes status. Three things must happen whenever a payment moves from Draft to Ready:

  • The Batch Module must check eligibility — only Ready payments can be added to a batch
  • The Export Module must exclude non-Ready payments from batch summaries
  • The Audit Log must record any status transition involving a batch assignment

Initially, PaymentController directly called BatchController and AuditLog inline. This tight coupling meant any change to batching or logging required editing PaymentController — violating the single responsibility principle.

Activity K2 — Pattern Applied

👁
Pattern Selected: Observer Pattern (Publish-Subscribe variant)

In FWAS, the Payment entity acts as the Subject. BatchController, AuditLogger, and ExportEngine register themselves as Observers of payment status changes. When PaymentController fires a status-change event, each Observer responds independently:

ObserverResponse to Payment Status Change
BatchControllerRe-evaluates whether the payment is eligible for batch inclusion (status must be Ready)
AuditLoggerWrites an immutable record to BatchAuditLog if the change involved a batch assignment or removal
ExportEngineMarks cached batch summaries as stale so the next export reflects updated payment state
SupplierSearchCacheOn new supplier registration, refreshes the search index so the payment form dropdown is immediately accurate
 
PaymentController (Subject)
   │
   │ notifyObservers(event: STATUS_CHANGED, paymentId, newStatus)
   │
   ├──▶ BatchController.onPaymentStatusChanged(paymentId, newStatus)
   ├──▶ AuditLogger.onPaymentStatusChanged(paymentId, newStatus)
   └──▶ ExportEngine.onPaymentStatusChanged(paymentId, newStatus)

SupplierController (Subject)
   │
   │ notifyObservers(event: SUPPLIER_REGISTERED, supplierId)
   │
   └──▶ SupplierSearchCache.onSupplierRegistered(supplierId)

Benefits for FWAS

  • PaymentController no longer imports or calls BatchController directly — coupling is broken
  • New observers (e.g., email notification to Supervisor) can be added without touching existing code
  • The AuditLog guarantee is enforced by the pattern, not by remembering to call a log function
  • Aligns with FWAS's audit compliance requirement — status changes cannot silently bypass the logger

Activity L1 — Build vs Buy

🔨
Decision: Build the core system; reuse open-source libraries for specific functions

No off-the-shelf product exists that solves the specific problem: preparing and batching supplier payment data for manual IFMS entry in an offline, single-institution environment. Commercial ERP systems require internet connectivity and are far beyond MUST's infrastructure scope. The build decision is also necessary because FWAS must integrate with MUST's specific workflow — IFMS export format, TIN validation rules, and the approval chain are all institution-specific.

Activity M2 — Evaluate Reuse

Benefits

  • PhpSpreadsheet saves significant dev time — Excel file generation involves complex binary formats
  • PHP PDO prepared statements provide SQL injection protection automatically
  • XAMPP gives a single installer that sets up the entire runtime, removing manual Apache/PHP configuration
  • Bootstrap ensures the interface works on different screen sizes without custom responsive CSS
  • Well-maintained libraries provide security patches when vulnerabilities are discovered

Challenges

  • PhpSpreadsheet requires Composer — must be available on deployment machine or bundled manually
  • Bootstrap must be kept as a local file copy to satisfy the offline requirement; a CDN link would fail
  • Version compatibility — PhpSpreadsheet 2.x requires PHP 8.1+; older PHP versions will not work
  • MIT licence attribution must be included for all reused components

Release Versioning

FWAS uses Semantic Versioning (MAJOR.MINOR.PATCH):

  • v1.0.0 — initial working release with all 22 use cases implemented
  • v1.1.0 — minor additions (e.g., Supervisor approval workflow UC-16)
  • v1.0.1 — bug fix releases

Database Change Management

Any change to the SQLite schema must be accompanied by a numbered SQL migration file stored in /database/migrations/. Example: 001_add_payment_purpose_column.sql. This ensures all team members and the deployed instance stay on the same schema version.

Activity P2 — Licensing Awareness

1. Attribution requirement (MIT Licence)

PhpSpreadsheet and Bootstrap are MIT-licenced, which requires that their copyright notice and licence text be included in the software. The FWAS codebase must include a /LICENCES/ folder containing the original licence files for both libraries. Omitting this is a licence violation even in non-commercial use.

2. No warranty clause

All licences used disclaim warranties. If PhpSpreadsheet produces a corrupt Excel file, the library authors bear no legal responsibility. FWAS must include its own data validation and testing to catch errors before they reach the Finance Officer.

3. Modification and distribution

If FWAS is ever distributed to other institutions beyond MUST, all bundled open-source components must still comply with their licence terms. MIT and Apache 2.0 both permit redistribution with attribution. No component used has a copyleft (GPL-style) licence that would force FWAS's own code to become open source.

4. XAMPP in production

XAMPP is designed primarily as a development environment. For long-term production deployment at MUST, the team recommends transitioning to a standalone Apache + PHP installation to avoid bundling unnecessary components (MariaDB, phpMyAdmin) that increase the attack surface on the finance workstation.

5. Dependency management

PhpSpreadsheet is installed via Composer. The composer.lock file must be committed to the repository so all team members and the deployment machine use identical library versions. Use composer install --no-dev on deployment to exclude development-only dependencies.
